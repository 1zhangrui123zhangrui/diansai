# 简易绳驱巡检装置 - 团队协作手册

## 一、仓库的意义和整体架构

### 1.1 这个仓库是干什么的

`diansai` 仓库是我们团队开发"简易绳驱巡检装置"电赛项目的**唯一代码和文档中心**。所有代码、文档、硬件配置、协议定义都在这里。任何人写的代码都必须通过这个仓库同步给其他人,**不准用 QQ/微信发压缩包**。

### 1.2 项目目录结构

```
diansai/
├── f407_main/      主控 STM32F407VET6,电机控制、坐标解算、串口屏、nRF 接收
├── f103_vision/    视觉前端 STM32F103C8T6,接收摄像头数据、nRF 发送
├── maixcam/        MaixCAM Pro 火源识别 Python 代码
├── screen/         淘晶驰 TJC4827X243 串口屏工程
├── shared/         两个单片机共用的头文件(通信协议定义)
├── docs/           题目、设计方案、标定记录、接线说明
├── .gitignore      忽略编译产物
└── README.md
```

### 1.3 硬件架构

```
           nRF24L01 无线(2.4GHz)
    ┌──────────────────────────────┐
    │                              │
┌───┴────────┐                ┌────┴─────────┐
│ 上平台     │                │ 下底座       │
│ F103C8T6   │                │ F407VET6     │
│  │         │                │  │           │
│  ├─UART─ MaixCAM Pro         │  ├─UART─ 串口屏(TJC4827X243)
│  │   (火源识别)              │  ├─PWM ── 4×步进电机驱动器(ZDT_X42S)
│  │                           │  ├─SPI── nRF24L01
│  └─SPI── nRF24L01            │  └─GPIO─ 蜂鸣器
│                              │
└──────────────────────────────┘
```

**通信流程**:

1. MaixCAM 识别火源 → UART → F103
2. F103 → nRF → F407
3. F407 → 电机 + 串口屏 + 蜂鸣器
4. 串口屏 → UART → F407(按键命令)

## 四、日常协作流程(必读)

### 4.1 核心规则

1. **main 分支只放稳定版**,不允许直接提交
2. **dev 分支是开发主线**,所有功能开发完合并到这里
3. **每个新功能从 dev 建一个新分支** `feat/模块名`,开发完合并回 dev
4. **每天开工前必须 pull**,收工前必须 push
5. **自己负责的模块外的文件不要乱改**
6. **main.c 和 .ioc 文件改动前先通知群里**,避免冲突

### 4.2 开发一个新功能的完整流程

假设你要开发 `motor.c`。

**步骤 1:切换到 dev 并更新**

- GitHub Desktop 顶部确认在 **dev** 分支
- 点 **Fetch origin** → 如果显示有新提交,点 **Pull origin**

**步骤 2:从 dev 建新分支**

- 顶部菜单 **Branch → New Branch**(Ctrl+Shift+N)
- **Name** 填:`feat/motor`
- **Based on** 保持 `dev`
- 点 **Create Branch**

创建后会自动切到 `feat/motor` 分支。

**步骤 3:写代码**

打开 Keil 工程,写 motor.c 和 motor.h,编译测试通过。

**步骤 4:提交**

回到 GitHub Desktop:

- 左边 **Changes** 区会列出你修改过的文件,确认都是你要提交的(不要提交 Keil 编译产物,如果出现 .o .hex .axf 这些说明 .gitignore 没生效,告诉队长)
- 左下角 **Summary** 填一句话描述:
    
    ```
    feat: 添加电机驱动,支持四路独立PWM
    ```
    
- **Description** 可以留空
- 点 **Commit to feat/motor**

**步骤 5:推送到 GitHub**

- 顶部点 **Publish branch**(第一次推送叫 Publish,之后叫 Push origin)

**步骤 6:发起 Pull Request(合并申请)**

- 顶部出现 **Create Pull Request** 按钮,点它,浏览器打开 GitHub 页面
- 确认 **base: dev** 和 **compare: feat/motor** 没错
- 点 **Create pull request**
- 在群里告诉队长:"feat/motor 分支好了,请 review"

**步骤 7:合并后清理本地**

队长 review 没问题后在 GitHub 网页上点 **Merge pull request**,然后告诉你可以回来了:

- GitHub Desktop 切回 dev 分支
- 点 **Pull origin** 把刚合并的代码拉下来
- 菜单 **Branch → Delete** 删掉本地的 feat/motor 分支(已合并的分支没用了)

### 4.3 每天开工的标准动作

```
打开 GitHub Desktop
  ↓
确认在 dev 分支
  ↓
Fetch origin
  ↓
如果有新提交,Pull origin
  ↓
切到你自己的 feat/xxx 分支继续写
```

### 4.4 每天收工的标准动作

```
写完代码,编译通过
  ↓
GitHub Desktop → Commit
  ↓
Push origin
```

哪怕功能还没写完,**当天的进度都要 push 上去**,一是备份,二是让别人能看到进度。

### 4.5 Commit message 写法

用英文前缀 + 中文描述:

```
feat: 添加电机驱动模块
fix: 修正 kinematics 中 Z 坐标符号错误
docs: 更新接线图
refactor: 把状态机从 main.c 移到 task.c
test: 添加 FK 算法测试点
chore: 更新 gitignore
```

前缀含义:`feat`=新功能,`fix`=修bug,`docs`=改文档,`refactor`=重构,`test`=测试,`chore`=杂项。

## 五、F407 主控配置说明

这一节解释 f407_main 工程在 CubeMX 里是怎么配的,以及为什么这么配。**修改前必须看懂**,否则会影响其他模块。

### 5.1 时钟配置

|项目|值|
|---|---|
|HSE 外部晶振|8 MHz|
|SYSCLK 系统时钟|168 MHz|
|HCLK|168 MHz|
|APB1 Timer clocks|84 MHz|
|APB2 Timer clocks|168 MHz|

**为什么 168MHz**:F407 的最高主频,性能最大化。TIM2~TIM5 都挂在 APB1 上,定时器时钟 84MHz,后面换算脉冲频率要用到。

### 5.2 引脚分配总表

#### 电机 PWM(四路独立定时器)

|功能|引脚|外设|
|---|---|---|
|M1 左前 脉冲|PA0|TIM2_CH1|
|M2 右前 脉冲|PA6|TIM3_CH1|
|M3 右后 脉冲|PB6|TIM4_CH1|
|M4 左后 脉冲|PA1|TIM5_CH2|

**为什么一个电机一个定时器**:因为四个电机需要**不同的脉冲频率**(运动学决定),同一个 TIM 的多个通道共享频率,做不到独立控制,所以必须分开。

**为什么 TIM5 用 CH2**:PA1 对应的是 TIM5_CH2,不是 CH1。用 CH2 避开了和其他引脚的冲突。

#### 电机方向和使能(GPIO)

|功能|引脚|标签|
|---|---|---|
|M1 方向|PE0|M1_DIR|
|M2 方向|PE1|M2_DIR|
|M3 方向|PE2|M3_DIR|
|M4 方向|PE3|M4_DIR|
|M1 使能|PE4|M1_ENA|
|M2 使能|PE5|M2_ENA|
|M3 使能|PE6|M3_ENA|
|M4 使能|PC13|M4_ENA|

**为什么 M4_ENA 用 PC13**:PE7 被蜂鸣器占了,所以借用 PC13。PC13 在 F407 上是普通 GPIO。

#### 串口

|功能|引脚|用途|
|---|---|---|
|USART1_TX|PA9|printf 调试|
|USART1_RX|PA10|printf 调试|
|USART2_TX|PD5|连串口屏 RX|
|USART2_RX|PD6|连串口屏 TX|

**为什么 USART2 用 PD5/PD6 而不是 PA2/PA3**:PA2 留作其他用途的备用,避免和 TIM5 等可能的映射冲突。

#### SPI1(接 nRF24L01)

|功能|引脚|标签|
|---|---|---|
|SPI1_SCK|PB3|-|
|SPI1_MISO|PB4|-|
|SPI1_MOSI|PB5|-|
|片选 CSN|PB1|NRF_CSN|
|使能 CE|PB0|NRF_CE|
|中断 IRQ|PB2|NRF_IRQ|

**为什么不用 PA5/PA6/PA7 默认 SPI1**:PA6 已经给 TIM3_CH1 用了(M2 脉冲),所以把 SPI1 重映射到 PB3/PB4/PB5。

**NRF_IRQ 配置**:外部中断,下降沿触发,内部上拉。对应的中断号是 EXTI line2。

#### 其他

|功能|引脚|标签|
|---|---|---|
|蜂鸣器|PE7|BUZZER|

### 5.3 外设参数

#### 四个电机 TIM2/3/4/5

四个定时器的参数完全一样:

|参数|值|含义|
|---|---|---|
|Prescaler (PSC)|83|84MHz ÷ 84 = 1MHz 计数频率,1 个计数 = 1us|
|Counter Period (ARR)|999|1MHz ÷ 1000 = 1kHz 初始 PWM 频率,代码中动态改|
|PWM Pulse (CCR)|500|50% 占空比|
|auto-reload preload|Enable|ARR 写入带缓冲,改频率平滑|
|Mode|PWM mode 1|向上计数,CNT<CCR 高电平|

**代码里怎么用**:

- 改转速:修改 ARR,公式 `频率 = 1MHz / (ARR + 1)`
- 改方向:写对应的 `MX_DIR` GPIO
- 启动/停止:`HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1)` / `Stop`
- 脉冲计数:使能 TIM 更新中断,每个 PWM 周期中断一次,软件累加计数

#### USART2(接屏)

- 波特率 115200
- 8 数据位,1 停止位,无校验
- 开启全局中断(接收按键命令用)

#### SPI1(接 nRF)

- Full-Duplex Master
- 8 bit data,MSB first
- Prescaler 8 → 实际速率约 10.5 MBit/s
- CPOL Low,CPHA 1 Edge
- NSS 软件控制

### 5.4 中断开启清单

这些中断在 NVIC 里必须是打勾状态,**改配置时不要关掉**:

- TIM2 global interrupt ✓
- TIM3 global interrupt ✓
- TIM4 global interrupt ✓
- TIM5 global interrupt ✓
- USART2 global interrupt ✓
- EXTI line2 interrupt ✓(nRF IRQ)

## 六、修改 CubeMX 配置的规则

### 6.1 谁可以改 .ioc

**只有一个人能改 .ioc 文件**,避免合并冲突。这个人叫"CubeMX 管理员",默认是队长。其他人需要增减引脚、改外设,**必须在群里提出需求**,由管理员统一改。

### 6.2 修改流程

1. 管理员在 dev 分支上从最新代码建一个 `chore/cubemx-xxx` 分支
2. 用 CubeMX 打开 `f407_main.ioc`,做修改
3. **Generate Code**,确认生成成功
4. Keil 打开工程,确认编译通过
5. commit + push,发 PR 合并回 dev
6. 合并后在群里通知:"CubeMX 更新了,大家 pull 一下"
7. 其他人 pull 前先 commit 自己的工作,避免冲突

## 七、编译工程

### 7.1 打开工程

用 Keil 打开 `f407_main/MDK-ARM/f407_main.uvprojx`,双击即可。

### 7.2 编译

- 快捷键 **F7** 编译
- 快捷键 **F8** 下载到板子
- 左上角下拉选 Target:默认 `f407_main`

### 7.3 常见问题

**问题 1:编译报 HAL_xxx 函数未定义**  
原因:HAL 库文件没加进工程。解决:CubeMX 重新 Generate Code,Keil 会自动更新工程配置。

**问题 2:下载时提示 No ST-LINK detected**  
原因:ST-Link 没接好,或驱动没装。解决:装 ST-Link 驱动,插紧 USB。

**问题 3:下载成功但程序不运行**  
原因:BOOT 跳线接错了。解决:BOOT0 接 GND,BOOT1 不管。

## 八、通信协议

所有协议常量定义在 `shared/protocol.h`,**两个单片机必须 include 同一个**,**不允许各自复制一份**。

### 8.1 串口屏 → F407

帧格式:`0x55 0x55 + CMD + [数据] + 0xFE`

|命令|宏定义|值|
|---|---|---|
|启动|CMD_START|0xA1|
|区域巡逻|CMD_AREA_PATROL|0xA2|
|顺序巡逻|CMD_SEQ_PATROL|0xA3|
|自动巡逻|CMD_AUTO_PATROL|0xA4|
|返回/急停|CMD_HOME|0xA0|
|零点标定|CMD_CALIBRATE|0xA5|

顺序巡逻命令后面跟 5 个 ASCII 字符('1'~'5'),最后 0xFE 结尾。

### 8.2 F103 → F407(nRF)

固定 4 字节,发送频率 10Hz:

```c
typedef struct {
    uint8_t head;         // 固定 0xAA
    uint8_t fire_flag;    // 0=无火, 1=有火
    uint8_t seq;          // 递增序号
    uint8_t checksum;     // head ^ fire_flag ^ seq
} VisionPacket_t;
```

### 8.3 F407 → 串口屏

ASCII 命令,末尾加 3 个 0xFF:

```
首页.va1.txt="20.20"0xFF 0xFF 0xFF    // 设置 X 坐标
首页.va2.txt="-15.30"0xFF 0xFF 0xFF   // 设置 Y 坐标
```

`va1/va2` 是首页上的全局变量控件,其他页面通过定时器 200ms 同步一次。

## 九、题目要求快速回顾

装置任务:通过控制四根绳子拉动悬吊模块(含摄像头和激光笔)在 60×60cm 平面内定位和巡检。

**基本要求**(50 分):

1. 按键启动,从任意初始位置回到中心
2. 四角橙色区域依次巡逻
3. 串口屏输入 5 个圆的巡逻顺序,按序依次到达
4. 实时显示模块坐标,精度 ±2cm

**发挥部分**(50 分):

1. 蛇形自动巡逻,间隔 10cm,覆盖整个巡检区
2. 检测到火源时停下,蜂鸣器响 3 声,继续巡逻
3. 记录火源坐标并显示在屏上(最多 2 个火源)

**五个橙色圆坐标**(中心为原点):

```
中心 (0, 0)
左下 (-20, -20)
右下 (20, -20)
右上 (20, 20)
左上 (-20, 20)
```

## 十、遇到问题怎么办

### 10.1 代码问题

- Git 相关问题:看本手册第四节
- CubeMX 配置问题:看本手册第五、六节
- 编译问题:看本手册第七节
- 都没解决的:发群里,描述清楚现象 + 截图

### 10.2 绝对不要做的事

- ❌ 在 main 分支直接提交代码
- ❌ 把队友的代码覆盖掉
- ❌ 没 pull 就开始写代码
- ❌ 把编译产物提交到仓库(.o .hex .axf .lst 等)
- ❌ 把本地 IDE 的个人配置提交(.vscode、.uvoptx 等)
- ❌ 改 .ioc 文件(除非你是 CubeMX 管理员)
- ❌ 写完代码不 push
- ❌ 用 QQ/微信传代码绕过仓库

### 10.3 遇到合并冲突

如果 push 时提示冲突:

1. 先 Pull origin
2. GitHub Desktop 会提示哪些文件冲突
3. 用记事本或 Keil 打开冲突文件,找到 `<<<<<<<` 和 `>>>>>>>` 标记
4. 手动决定保留哪段代码,删掉标记
5. 保存后 Commit,再 Push
6. 搞不定就发群里喊人来帮忙,**不要乱改**
